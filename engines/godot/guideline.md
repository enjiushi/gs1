# engines/godot/

Godot-specific client scaffolding for the current first-party visual runtime path, with the native adapter now centered on `godot-cpp` classes that own Godot-side scene manipulation while the gameplay DLL stays behind the shared host/session bridge.

## Usage
- Read this file before scanning the folder in detail.
- If you open a child folder, read its `guideline.md` first when present.
- When files or folders in this directory change, update this file in the same change.

## Contents
- `native/`: Godot runtime-extension native source files built by the repo CMake configuration, including the director-owned pure C++ adapter service that hosts the gameplay runtime and lightweight remaining cue dispatcher, the bootstrap `Gs1GodotDirectorControl` that swaps dedicated menu/regional/site scenes by projected app state, keeps the loading screen up through threaded load completion plus a short configurable hidden-frame reveal delay so scene-tree rebuild/setup hitches stay masked during expensive transitions, the split scene/UI/controller roots that separate world, shell, and panel ownership, scene-owned direct-subscriber controllers for status, site HUD, inventory, phone, craft, tasks, regional HUD/selection/summary, overlay state, and the regional tech tree, with those pure panel controllers now acting as message-appliers that mutate only the UI state named by authoritative gameplay messages instead of running per-frame refresh or dirty-loop behavior, the craft panel now reading contextual craft options and placement preview state directly from `Gs1GameStateView`, the main-menu and site-session shells now owning their own authored button wiring instead of relying on the removed shared action-panel controller, the director now explicitly queued-preloading the current regional tech-tree icon set while the regional tech-tree controller also prewarms its icon and tooltip caches during regional-map initialization so the first overlay open avoids lazy resource IO, and the site-scene transition now simply keeps gameplay ticking through `SITE_LOADING` while the async-loaded site scene instantiates without any adapter-side message buffering or `SITE_SCENE_READY` handshake, while the site-scene-owned Godot presenter now rebuilds from state-view and tile-query reads on coarse adapter-local notifications rather than gameplay snapshot families, the site inventory controller now reconciling authored worker-pack and opened-storage sub-panels instead of one flat inline inventory block, the adapter service now optionally exposing a loopback-only debug HTTP control port that reuses semantic host/UI actions instead of synthetic widget clicks, and the regional map still relying on authored backdrop and site-marker scenes, reconciling its world view from campaign state-view reads while consuming blank world clicks through a shared pure-C++ policy helper, and feeding the tabbed research overlay.
- `project/`: The actual Godot project shell, including the bootstrap director scene plus the dedicated main-menu, regional-map, and site-session scene assets, the reusable site HUD scene, the authored regional-map dune-basin backdrop and camp marker scenes, the authored player-meter, phone, worker-pack, and opened-storage panel sub-scenes used by the site-session shell, the runtime extension packaging files, and the engine-specific progression/content/render-resource TOML files that map gameplay tech ids plus unlockable, content, modifier, task-template, reward-candidate, and site ids to Godot icon, thumbnail, and scene asset paths.
