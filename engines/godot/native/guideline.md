# engines/godot/native/

Native C++ source for the Godot runtime extension target, now owned by a `godot-cpp`-first adapter that talks to the engine through normal `godot-cpp` classes and APIs while keeping the gameplay DLL boundary in the shared host/session layer. Pure panel controllers in this folder should behave as message-appliers: setup happens in lifecycle entry points such as `_ready()`, gameplay messages mutate only the affected UI state immediately, `_process()` is typically just an empty placeholder, and full keyed reconciliation is reserved for authoritative snapshot-boundary messages instead of per-frame refresh loops.

## Usage
- Read this file before scanning the folder in detail.
- When files or folders in this directory change, update this file in the same change.

## Contents
- `gs1_godot_extension_entry.cpp`: `godot-cpp` extension entry point that registers `Gs1GodotDirectorControl` plus the split scene and UI controller roots for the main-menu/regional-map/site-session shells through the normal binding bootstrap.
- `gs1_godot_adapter_service.h` / `gs1_godot_adapter_service.cpp`: Pure C++ adapter service that owns the gameplay runtime session, loads the staged runtime/config roots, drains engine messages directly from the DLL queue, dispatches them through per-message subscriber buckets, forwards host-event submissions back into gameplay, now buffers all drained engine messages while the director-owned loading transition is active and replays them in order after the target scene subscribes, and still asserts in debug builds if any snapshot-style projection family is subscribed by more than one controller.
- `gs1_godot_debug_http_protocol.h` / `gs1_godot_debug_http_protocol.cpp`: Pure C++ parser for the optional loopback debug HTTP control contract, translating the visual-smoke-style `/ui-action`, `/site-action`, `/site-context`, inventory, and movement payloads into queueable semantic adapter commands without depending on Godot.
- `gs1_godot_debug_http_server.h` / `gs1_godot_debug_http_server.cpp`: Lightweight Windows loopback HTTP server used only for optional debug automation in the Godot adapter, accepting semantic control POSTs on a background thread and deferring all gameplay submission back to the adapter-service frame thread.
- `gs1_godot_controller_context.h` / `gs1_godot_controller_context.cpp`: Small controller lookup helpers that let scene/UI/presenter nodes find the active `Gs1GodotDirectorControl` and its shared adapter service without hard-wiring controller-to-controller references.
- `gs1_godot_projection_types.h`: Shared Godot-adapter projection structs and aliases used by the adapter service, scene controllers, and panel controllers without forcing them to include one centralized main-screen cache header.
- `gs1_godot_status_panel_controller.h` / `gs1_godot_status_panel_controller.cpp`: Scene-owned panel controller for the status feed label, owning only its local campaign/action summary state plus the last submitted local action string and rebuilding the label directly from subscribed messages instead of polling runtime-link state each frame.
- `gs1_godot_overlay_panel_controller.h` / `gs1_godot_overlay_panel_controller.cpp`: Small direct-subscriber UI controller for the overlay panel, owning the currently presented overlay mode, projected protection-selector and site-result modal setup reconciliation, the local overlay-mode button wiring, and direct label/visibility updates straight from the message stream without routing through a shared main-screen cache.
- `gs1_godot_regional_summary_panel_controller.h` / `gs1_godot_regional_summary_panel_controller.cpp`: Direct-subscriber controller for the regional map summary/feed labels, now consuming its own dedicated regional-summary snapshot family alongside campaign-resource messages instead of sharing the regional scene snapshot stream.
- `gs1_godot_site_summary_panel_controller.h` / `gs1_godot_site_summary_panel_controller.cpp`: Direct-subscriber controller for the site summary/title labels, now consuming its own dedicated site-summary snapshot family plus HUD/action/weather updates instead of borrowing data from the status controller.
- `gs1_godot_site_hud_controller.h` / `gs1_godot_site_hud_controller.cpp`: Scene-owned direct-subscriber controller for the site-session HUD scene, owning projected player-meter/cash/completion labels, top-level site panel launcher buttons, and small badge state while submitting HUD intents through the adapter service, with HUD text and badges applied immediately from gameplay messages plus a dedicated site-HUD storage snapshot family.
- `gs1_godot_inventory_panel_controller.h` / `gs1_godot_inventory_panel_controller.cpp`: Direct-subscriber controller for the inventory panel, locally assembling only the worker-pack and opened-storage presentation state required for that panel, rebuilding its rows only when subscribed messages change that state, deriving panel visibility from authoritative inventory/open-storage messages instead of app-state gating, and now consuming a dedicated site-inventory snapshot family rather than the shared site snapshot stream.
- `gs1_godot_phone_panel_controller.h` / `gs1_godot_phone_panel_controller.cpp`: Scene-owned direct-subscriber controller for the phone panel, owning the section summary label, phone-tab plus close-button input wiring, dynamic phone-listing button reconciliation, and phone intent submission while deriving visibility directly from projected phone-open state, with listings now arriving through a dedicated phone-panel snapshot family.
- `gs1_godot_craft_panel_controller.h` / `gs1_godot_craft_panel_controller.cpp`: Scene-owned direct-subscriber controller for the craft/placement panel, owning the craft/placement summary label, dynamic craft-option button reconciliation, and craft-request submission while deriving panel visibility from projected craft/placement message state instead of screen app-state.
- `gs1_godot_task_panel_controller.h` / `gs1_godot_task_panel_controller.cpp`: Direct-subscriber controller for the site task/modifier panel, owning the task-summary label and disabled task/modifier row reconciliation plus its local icon/metadata cache, with panel visibility driven by authoritative UI-surface visibility messages rather than shell app-state, and tasks/modifiers now projected through a dedicated task-panel snapshot family.
- `gs1_godot_action_panel_controller.h` / `gs1_godot_action_panel_controller.cpp`: Scene-owned direct-subscriber controller for fixed-slot and projected site-action wiring, now owning only the shared action-panel UI-panel family for fixed-slot actions plus the static protection-overlay opener and site control wiring instead of also subscribing the overlay setup stream.
- `gs1_godot_regional_map_hud_controller.h` / `gs1_godot_regional_map_hud_controller.cpp`: Direct-subscriber controller for the regional-map HUD scene, owning campaign summary text, selected-site hint text, and the tech-panel entry button so future regional-map HUD buttons can be added without routing those panel-local controls through the shell, with the HUD itself simply living while the regional scene exists, applying selected-site and campaign-summary labels only when the dedicated regional-map-HUD snapshot commits or campaign resources change, and deriving the tech button state from surface visibility instead of the shared panel snapshot.
- `gs1_godot_regional_selection_panel_controller.h` / `gs1_godot_regional_selection_panel_controller.cpp`: Direct-subscriber controller for the regional selection panel, owning the selected-site brief plus dedicated support-feed and provided-loadout subviews alongside deployment action-button reconciliation while following its dedicated regional-selection snapshot family and dedicated regional-selection UI-panel family, with full site-brief reconciliation reserved for authoritative selection-snapshot completion while UI-panel completion/close messages only refresh the action/support/loadout subviews, with the projected loadout now rendered as a read-only inventory-style slot grid with item icons instead of a flat text summary, and with the panel now suppressing the clear-selection action button so deselection happens by blank-map click while the panel keeps only the deploy affordance.
- `gs1_godot_regional_tech_tree_panel_controller.h` / `gs1_godot_regional_tech_tree_panel_controller.cpp`: Scene-owned direct-subscriber controller for the regional research overlay, owning its local progression-view state, the unlockable/technology card reconciliation, the close-button plus card-action input wiring, icon/tooltip caches, and tech action-button routing, with overlay visibility driven by authoritative progression/surface messages instead of app-state checks, the authored ladder layout now built locally from shared technology/unlock definitions while runtime progression messages only refresh dynamic per-entry state, and the ladder now splitting full research-card buttons from lightweight non-interactive marker/placeholder controls so non-card rows no longer allocate either the full card subtree or button widgets on first open.
- `gs1_godot_director_control.h` / `gs1_godot_director_control.cpp`: Bootstrap `godot-cpp` `Control` that owns the shared adapter service, watches the authoritative projected app state, swaps the menu/regional/site scene instance without synthesizing bootstrap gameplay messages, and now routes the main-menu-to-regional-map transition through a lightweight loading scene while Godot threaded-loads the full regional-map `PackedScene`, with the director also owning the current explicit regional tech-tree icon preload list, driving the loading-scene progress bar from the real threaded scene/preload work, and buffering/replaying all engine messages during that loading phase so controller initialization still happens only when the complete target scene, queued icon textures, and deferred engine-message batch are ready.
- `gs1_godot_main_menu_ui_controller.h` / `gs1_godot_main_menu_ui_controller.cpp`: Main-menu UI root that owns the shell-local button wiring and start/quit runtime submissions without carrying any regional-map or site-session responsibilities.
- `gs1_godot_regional_map_scene_controller.h` / `gs1_godot_regional_map_scene_controller.cpp`: Regional-map scene root that owns world-node lookup, authoritative keyed regional snapshot reconciliation at `END_REGIONAL_MAP_SNAPSHOT`, authored camp marker scene instancing, camera framing, runtime state tinting for authored marker accents, and regional click-picking/input dispatch while leaving backdrop art, HUD, and panel behavior to authored scenes plus dedicated UI controllers instead of generating terrain, links, or fallback marker geometry in C++, with the shared regional-map snapshot family now reserved to this controller as the sole subscriber and no longer carrying an empty per-frame `_process()` tick, with blank-world clicks now submitting clear-selection when nothing was picked, with selected visuals no longer falling back to the first site when gameplay says nothing is selected, and with the authored marker scene now exposing a dedicated selection ring node that the controller toggles through stable keyed reconciliation.
- `gs1_godot_regional_map_ui_controller.h` / `gs1_godot_regional_map_ui_controller.cpp`: Regional-map UI root that owns only shell-level UI-surface visibility handling for the selection shell, summary shell, and shared tech-tree overlay while the regional scene exists, without app-state gating or panel-local subscriptions.
- `gs1_godot_site_scene_controller.h` / `gs1_godot_site_scene_controller.cpp`: Site-session scene root and world presenter that owns the bootstrap camera/light, projected tile/plant/device/worker visuals, gameplay-ID-to-Godot-instance reconciliation, and direct runtime-message subscription for site rendering without a separate site-view controller node, now consuming its own dedicated site-scene snapshot family instead of sharing the broad site snapshot stream with panel controllers.
- `gs1_godot_site_session_ui_controller.h` / `gs1_godot_site_session_ui_controller.cpp`: Site-session UI root that still owns the deferred local selected-tile path, static site command wiring, controller-local site snapshot assembly for those shell interactions, and the small remaining shell-level host submissions without owning or subscribing the scene-owned status, HUD, inventory, phone, craft, task, action, overlay, summary, or tech-tree panel controllers.
- `godot_progression_resources.h` / `godot_progression_resources.cpp`: Lightweight Godot-only TOML loader for progression and broader render resources, keyed by shared `tech_node_id`, unlockable content kind/id pairs, general content kind/id pairs, and domain/id entries such as modifiers, task templates, reward candidates, and sites so the adapter can resolve icon, thumbnail, and scene paths without moving shared descriptions or gameplay meaning out of the gameplay-authored content tables, now loading from the staged `res://gs1/godot_config/` package copy.
