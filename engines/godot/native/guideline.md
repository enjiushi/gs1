# engines/godot/native/

Native C++ source for the Godot runtime extension target, now owned by a `godot-cpp`-first adapter that talks to the engine through normal `godot-cpp` classes and APIs while keeping the gameplay DLL boundary in the shared host/session layer.

## Usage
- Read this file before scanning the folder in detail.
- When files or folders in this directory change, update this file in the same change.

## Contents
- `gs1_godot_extension_entry.cpp`: `godot-cpp` extension entry point that registers `Gs1RuntimeNode`, `Gs1SiteViewNode`, `Gs1GodotMainScreenControl`, and the bootstrap scene router at scene-init time through the normal binding bootstrap.
- `gs1_godot_main_screen_control.h`: `godot-cpp` `Control` declaration for the routed per-screen shell instances that consume gameplay projection state and own the menu/regional/site HUD surfaces inside the state-specific Godot scenes.
- `gs1_godot_main_screen_control.cpp`: Native `godot-cpp` screen-shell implementation that reads the runtime node directly in C++, consumes typed `Dictionary` projection data for the HUD/site panels, drives the menu/regional-map/site UI, reconciles regional terrain/link/site nodes and projected action/listing/craft buttons by stable gameplay keys instead of clearing whole Godot containers on revision changes, now binds authored main-menu and regional-map fixed buttons by editor-authored panel-plus-slot metadata, tracks selected site/tile state, forwards gameplay host actions without relying on GDScript or `Object::call`, now leaves the menu and regional-map art composition to authored Godot backdrop scenes instead of generated box terrain, routes regional site markers through site-id-driven scene config instead of a hardcoded marker scene, routes task-template and modifier rows through adapter-owned resource icons, routes the tech-tree projection into the overlay tabs, faction-tab strip, and scrollable card grids with adapter-owned descriptions sourced from the staged `gs1/runtime/adapter_metadata/` package copy, clickable faction sub-tabs, and composite lock-badged cards that keep hover text available even while locked cards remain non-actionable, while also reading engine-specific content icon bindings for item-heavy inventory/phone/craft/placement surfaces and guarding regional-map world picking so HUD and overlay controls keep their own left-click input instead of being intercepted as site-selection clicks.
- `godot_progression_resources.h` / `godot_progression_resources.cpp`: Lightweight Godot-only TOML loader for progression and broader render resources, keyed by shared `tech_node_id`, unlockable content kind/id pairs, general content kind/id pairs, and domain/id entries such as modifiers, task templates, reward candidates, and sites so the adapter can resolve icon, thumbnail, and scene paths without moving shared descriptions or gameplay meaning out of the gameplay-authored content tables, now loading from the staged `res://gs1/godot_config/` package copy.
- `gs1_godot_runtime_node.h`: `godot-cpp` `Node` declaration that owns the shared gameplay runtime session, resolves both the gameplay DLL path and the staged runtime package root, packages Godot-owned config into the shared adapter-config startup payload, exposes typed projection-query helpers plus a full `Dictionary` projection surface, tracks a monotonic projection revision for adapter cache invalidation, and forwards real host-event submissions from the Godot client shell through direct native calls.
- `gs1_godot_runtime_node.cpp`: `godot-cpp` runtime-loop plumbing for `Gs1RuntimeNode`, including shared host-session startup against the staged `res://gs1/runtime/` package root, adapter-side metadata catalog loading from the staged `adapter_metadata/` copy, shared-projection-cache integration, full menu/regional-map/site `Dictionary` projection building, revision bumps for runtime-state changes, and host-event submission methods, now routing Godot adapter config through the shared runtime initialization path instead of treating the adapter as a special-case config reader.
- `gs1_godot_scene_router_control.h`: `godot-cpp` `Control` declaration for the bootstrap scene router that watches the authoritative projected app state and swaps the menu, regional-map, or site-session scene instance while keeping the shared runtime node alive.
- `gs1_godot_scene_router_control.cpp`: Native router implementation that instantiates the state-specific `.tscn` files, wires them back to the persistent runtime node, and keeps site-view runtime lookup working across routed scene boundaries.
- `gs1_godot_site_view_node.h`: Native `Node3D` presenter declaration that maps gameplay-authored entity or visual IDs to Godot `ObjectID`s so the adapter owns stable Godot scene objects for tiles, plants, devices, and the worker marker through persistent per-ID registries, now with an explicit runtime-node path for routed scene setups.
- `gs1_godot_site_view_node.cpp`: Native presenter implementation that creates the bootstrap camera/light, owns real Godot mesh nodes for site visuals, consumes gameplay-authored snapshot and delta messages once from the runtime node, reconciles site tiles, worker state, and plant/device visuals across authoritative snapshots by persistent gameplay IDs instead of tearing them down at snapshot begin, and now allows plant/structure visuals to resolve optional engine-owned scene resources by shared content id before falling back to native placeholder meshes.
