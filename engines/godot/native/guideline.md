# engines/godot/native/

Native C++ source for the Godot runtime extension target, now owned by a `godot-cpp`-first adapter that talks to the engine through normal `godot-cpp` classes and APIs while keeping the gameplay DLL boundary in the shared host/session layer.

## Usage
- Read this file before scanning the folder in detail.
- When files or folders in this directory change, update this file in the same change.

## Contents
- `gs1_godot_extension_entry.cpp`: `godot-cpp` extension entry point that registers `Gs1RuntimeNode`, `Gs1SiteViewNode`, `Gs1GodotMainScreenControl`, and the bootstrap scene router at scene-init time through the normal binding bootstrap.
- `gs1_godot_projection_types.h`: Shared Godot-adapter projection structs and aliases used by the runtime node, site view, reducers, and panel controllers without forcing them to include one centralized main-screen cache header.
- `gs1_godot_panel_state_reducers.h` / `gs1_godot_panel_state_reducers.cpp`: Message-driven reducer helpers that now split gameplay projection assembly by responsibility such as status, UI setups, UI panels, progression views, regional map, and site session state, so live message-derived state can move into the specific controller that owns each UI surface instead of one global main-screen cache.
- `gs1_godot_status_panel_controller.h` / `gs1_godot_status_panel_controller.cpp`: First extracted panel controller owning the status feed label, its reducer-backed app/campaign/action summary state, and self-refresh logic, with the shell only feeding runtime-link plus last-submitted-action strings into that controller instead of orchestrating panel redraws.
- `gs1_godot_overlay_panel_controller.h` / `gs1_godot_overlay_panel_controller.cpp`: Small direct-subscriber UI controller for the overlay panel, owning the currently presented overlay mode, site-state panel visibility, the local overlay-mode button wiring, and direct label refresh straight from the message stream without routing through a shared main-screen cache.
- `gs1_godot_regional_summary_panel_controller.h` / `gs1_godot_regional_summary_panel_controller.cpp`: Direct-subscriber controller for the regional map summary/feed labels, owning only the campaign-resource plus regional-map presentation state it needs and no longer reading another controller's data.
- `gs1_godot_site_summary_panel_controller.h` / `gs1_godot_site_summary_panel_controller.cpp`: Direct-subscriber controller for the site summary/title labels, owning only the site summary presentation state it needs from site, HUD, and action messages instead of borrowing data from the status controller.
- `gs1_godot_inventory_panel_controller.h` / `gs1_godot_inventory_panel_controller.cpp`: Direct-subscriber controller for the inventory panel, locally assembling only the worker-pack and opened-storage presentation state required for that panel and now also owning that panel's site-state visibility.
- `gs1_godot_phone_panel_controller.h` / `gs1_godot_phone_panel_controller.cpp`: Direct-subscriber controller for the phone panel, owning the section summary label, site-state panel visibility, the phone-tab plus close-button input wiring, and dynamic phone-listing button reconciliation while delegating phone intents back through one injected callback instead of bouncing those panel-local presses through the main screen shell.
- `gs1_godot_craft_panel_controller.h` / `gs1_godot_craft_panel_controller.cpp`: Direct-subscriber controller for the craft/placement panel, owning the craft/placement summary label, site-state panel visibility, and dynamic craft-option button reconciliation while resolving pressed options back through an injected submit callback instead of leaving that list in the main screen shell.
- `gs1_godot_task_panel_controller.h` / `gs1_godot_task_panel_controller.cpp`: Direct-subscriber controller for the site task/modifier panel, owning site-state panel visibility, the task-summary label, and the disabled task/modifier row reconciliation plus its local icon/metadata cache instead of leaving that panel in the main screen shell.
- `gs1_godot_action_panel_controller.h` / `gs1_godot_action_panel_controller.cpp`: Direct-subscriber controller for fixed-slot and projected site-action wiring, owning the UI-setup and UI-panel reducer slices needed to bind slot actions, the static protection-overlay opener, and dynamic site control buttons while forwarding presses through injected UI-action submission callbacks.
- `gs1_godot_regional_map_hud_controller.h` / `gs1_godot_regional_map_hud_controller.cpp`: Direct-subscriber controller for the regional-map HUD scene, owning campaign summary text, selected-site hint text, and the tech-panel entry button so future regional-map HUD buttons can be added without routing those panel-local controls through the shell.
- `gs1_godot_regional_selection_panel_controller.h` / `gs1_godot_regional_selection_panel_controller.cpp`: Direct-subscriber controller for the regional selection panel, owning the selected-site brief plus dedicated support-feed and provided-loadout summaries alongside deployment action-button reconciliation while following authoritative regional-map messages for selected-site state instead of accepting shell-pushed selection, now also keeping those panel-local button signals inside the controller instead of routing them back through the shell.
- `gs1_godot_regional_tech_tree_panel_controller.h` / `gs1_godot_regional_tech_tree_panel_controller.cpp`: Direct-subscriber controller for the regional research overlay, owning the progression-view reducer, overlay/panel visibility, the unlockable/technology card reconciliation, the close-button plus card-action input wiring, icon/tooltip caches, and tech action-button routing instead of leaving the tech-tree overlay in the main screen shell.
- `gs1_godot_main_screen_control.h`: `godot-cpp` `Control` declaration for the routed per-screen shell instances that subscribe to the runtime node's authoritative engine-message dispatcher, now narrowed toward a wiring/input/world shell instead of a UI-state hub.
- `gs1_godot_main_screen_control.cpp`: Native `godot-cpp` screen-shell implementation that still coordinates top-level menu/regional/site presentation and shell-owned world reconciliation, but now leaves panel refresh, panel visibility, and panel-local button wiring to the direct-subscriber controllers while preserving routed scene wiring, native gameplay host calls, and only a comparatively small generic shell input/submit layer for scene-level controls, including the stronger regional-map selected-site ring/hint presentation over the authored desert terrain shell and a dedicated regional-map HUD controller entry point.
- `godot_progression_resources.h` / `godot_progression_resources.cpp`: Lightweight Godot-only TOML loader for progression and broader render resources, keyed by shared `tech_node_id`, unlockable content kind/id pairs, general content kind/id pairs, and domain/id entries such as modifiers, task templates, reward candidates, and sites so the adapter can resolve icon, thumbnail, and scene paths without moving shared descriptions or gameplay meaning out of the gameplay-authored content tables, now loading from the staged `res://gs1/godot_config/` package copy.
- `gs1_godot_runtime_node.h`: `godot-cpp` `Node` declaration that owns the shared gameplay runtime session, resolves both the gameplay DLL path and the staged runtime package root, packages Godot-owned config into the shared adapter-config startup payload, and acts as the sole adapter-side drain point plus typed engine-message dispatcher for subscribed presenters before forwarding real host-event submissions from the Godot client shell through direct native calls.
- `gs1_godot_runtime_node.cpp`: `godot-cpp` runtime-loop plumbing for `Gs1RuntimeNode`, including shared host-session startup against the staged `res://gs1/runtime/` package root, adapter-side metadata catalog loading from the staged `adapter_metadata/` copy, single-point engine-message dequeue plus subscriber dispatch with reset notifications, and host-event submission methods, now routing Godot adapter config through the shared runtime initialization path instead of treating the adapter as a special-case config reader.
- `gs1_godot_scene_router_control.h`: `godot-cpp` `Control` declaration for the bootstrap scene router that watches the authoritative projected app state and swaps the menu, regional-map, or site-session scene instance while keeping the shared runtime node alive.
- `gs1_godot_scene_router_control.cpp`: Native router implementation that instantiates the state-specific `.tscn` files, wires them back to the persistent runtime node, seeds new routed screen shells with the current app state before later projection messages arrive, keeps site-view runtime lookup working across routed scene boundaries, unsubscribes the outgoing scene before queueing it for deletion, and now switches immediately on authoritative `SET_APP_STATE` engine messages instead of waiting for a later process pass or polling shared projection state.
- `gs1_godot_site_view_node.h`: Native `Node3D` presenter declaration that maps gameplay-authored entity or visual IDs to Godot `ObjectID`s so the adapter owns stable Godot scene objects for tiles, plants, devices, and the worker marker through persistent per-ID registries, now with an explicit runtime-node path for routed scene setups.
- `gs1_godot_site_view_node.cpp`: Native presenter implementation that creates the bootstrap camera/light, owns real Godot mesh nodes for site visuals, subscribes only to the site snapshot/tile/worker/visual engine messages it cares about, reconciles site tiles, worker state, and plant/device visuals across authoritative snapshots by persistent gameplay IDs instead of tearing them down at snapshot begin, resets cleanly when the runtime node signals a message-stream reset, and now allows plant/structure visuals to resolve optional engine-owned scene resources by shared content id before falling back to native placeholder meshes.
