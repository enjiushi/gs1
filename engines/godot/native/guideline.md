# engines/godot/native/

Native C++ source for the Godot runtime extension target, now owned by a `godot-cpp`-first adapter that talks to the engine through normal `godot-cpp` classes and APIs while keeping the gameplay DLL boundary in the shared host/session layer. Pure panel controllers in this folder should behave as message-appliers: setup happens in lifecycle entry points such as `_ready()`, gameplay messages mutate only the affected UI state immediately, `_process()` is typically just an empty placeholder, and full keyed reconciliation is reserved for authoritative snapshot-boundary messages instead of per-frame refresh loops.

## Usage
- Read this file before scanning the folder in detail.
- When files or folders in this directory change, update this file in the same change.

## Contents
- `gs1_godot_extension_entry.cpp`: `godot-cpp` extension entry point that registers `Gs1GodotDirectorControl` plus the split scene and UI controller roots for the main-menu/regional-map/site-session shells through the normal binding bootstrap.
- `gs1_godot_adapter_service.h` / `gs1_godot_adapter_service.cpp`: Pure C++ adapter service that owns the gameplay runtime session, loads the staged runtime/config roots, drains engine messages directly from the DLL queue, dispatches them through per-message subscriber buckets, forwards host-event submissions back into gameplay, and now runs the Godot adapter frame as `queue/send host events -> DLL phase 1 -> drain/apply phase-1 messages -> let Godot-side reactions queue/send more events -> DLL phase 2`, intentionally leaving phase-2 output queued inside the DLL until the next frame's drain instead of applying it immediately, while still buffering all drained engine messages during the director-owned regional loading transition and replaying them in order after the target scene subscribes, now exposing an explicit `submit_site_scene_ready()` host-event helper so site bootstrap stays blocked until the Godot scene is actually ready, synthesizing coarse `SET_APP_STATE` plus `PRESENTATION_DIRTY_ALL` notifications directly from polled `Gs1GameStateView` reads after startup and phase 1 so the native host no longer depends on gameplay-owned projection dirty caches, and now also owning the host-local phone section/open state, inventory panel session state, and protection overlay session state that the gameplay DLL no longer stores authoritatively, with the old exclusive-snapshot subscriber guard now effectively dormant because the native adapter no longer consumes snapshot families, and now also tripping a debug assertion before falling back to the normal runtime-reset path when an unexpected adapter/runtime-session failure occurs so those bugs no longer masquerade as ordinary main-menu returns.
- `gs1_godot_engine_message_policy.h`: Header-only Godot adapter policy helper that still fronts the exclusive-subscriber check path, but it now returns `false` for every message because the native adapter no longer consumes any snapshot-opening families.
- `gs1_godot_debug_http_protocol.h` / `gs1_godot_debug_http_protocol.cpp`: Pure C++ parser for the optional loopback debug HTTP control contract, translating the visual-smoke-style `/ui-action`, `/site-action`, `/site-context`, inventory, and movement payloads into queueable semantic adapter commands without depending on Godot.
- `gs1_godot_debug_http_server.h` / `gs1_godot_debug_http_server.cpp`: Lightweight Windows loopback HTTP server used only for optional debug automation in the Godot adapter, accepting semantic control POSTs on a background thread and deferring all gameplay submission back to the adapter-service frame thread.
- `gs1_godot_director_scene_policy.h`: Header-only director scene-switch policy shared between the Godot director and the pure C++ regression test so same-screen app-state changes such as `REGIONAL_MAP -> SITE_LOADING` do not rebuild the active scene and duplicate exclusive snapshot subscribers.
- `gs1_godot_regional_map_click_policy.h`: Header-only regional-map input policy shared between the scene controller and a pure C++ regression test so blank-map clicks are still consumed even when they do not change selection, locked/completed marker hits stay consumed without submitting invalid deployment-selection actions, and unhandled clicks that still land inside regional-map-owned UI are swallowed before they can leak to unrelated handlers.
- `gs1_godot_regional_site_selection_policy.h`: Header-only regional-map selection helper shared between controllers and a pure C++ regression test so regional HUD/summary/panel state only shows a selected site when gameplay explicitly projected one and never falls back to Site 1 or another first revealed marker.
- `gs1_godot_controller_context.h` / `gs1_godot_controller_context.cpp`: Small controller lookup helpers that let scene/UI/presenter nodes find the active `Gs1GodotDirectorControl` and its shared adapter service without hard-wiring controller-to-controller references.
- `gs1_godot_projection_types.h`: Shared Godot-adapter projection structs and aliases used by the adapter service, scene controllers, and panel controllers without forcing them to include one centralized main-screen cache header.
- `gs1_godot_status_panel_controller.h` / `gs1_godot_status_panel_controller.cpp`: Scene-owned panel controller for the status feed label, owning only its local campaign/action summary state plus the last submitted local action string and rebuilding the label directly from subscribed messages instead of polling runtime-link state each frame.
- `gs1_godot_overlay_panel_controller.h` / `gs1_godot_overlay_panel_controller.cpp`: Small direct-subscriber UI controller for the overlay panel, owning the currently presented overlay mode, projected protection-selector and site-result modal setup reconciliation, the local overlay-mode button wiring, and direct label/visibility updates straight from the message stream without routing through a shared main-screen cache.
- `gs1_godot_regional_summary_panel_controller.h` / `gs1_godot_regional_summary_panel_controller.cpp`: Direct-subscriber controller for the regional map summary/feed labels, now consuming its own dedicated regional-summary snapshot family alongside campaign-resource messages instead of sharing the regional scene snapshot stream, summarizing blocked-versus-available prototype sites now that all four site markers stay visible from campaign start, and leaving the selected-site line empty when gameplay has not explicitly selected a site.
- `gs1_godot_site_summary_panel_controller.h` / `gs1_godot_site_summary_panel_controller.cpp`: Direct-subscriber controller for the site summary/title labels, now consuming its own dedicated site-summary snapshot family plus HUD/action/weather updates instead of borrowing data from the status controller.
- `gs1_godot_site_hud_controller.h` / `gs1_godot_site_hud_controller.cpp`: Scene-owned direct-subscriber controller for the site-session HUD scene, owning the authored player-meter/cash/completion sub-scene plus the top-level site panel launcher buttons and small badge state while submitting HUD intents through the adapter service, with HUD text and badges still applied from semantic gameplay messages while worker-pack lookup now refreshes from the exported gameplay-state view on coarse presentation-dirty signals instead of a dedicated HUD storage snapshot stream.
- `gs1_godot_inventory_panel_controller.h` / `gs1_godot_inventory_panel_controller.cpp`: Direct-subscriber controller for the inventory panel, locally assembling only the worker-pack and opened-storage presentation state required for that panel, now reconciling separate authored worker-pack and opened-storage sub-panels while rebuilding their slot rows from the exported gameplay-state view on coarse presentation-dirty signals, deriving panel visibility from authoritative inventory/open-storage messages instead of app-state gating, and no longer consuming a dedicated inventory-panel snapshot family.
- `gs1_godot_phone_panel_controller.h` / `gs1_godot_phone_panel_controller.cpp`: Scene-owned direct-subscriber controller for the phone panel, owning the section summary label, phone-tab plus close-button input wiring, dynamic phone-listing button reconciliation, and phone intent submission while deriving visibility directly from projected phone-open state, with listings now refreshed from the exported gameplay-state view on coarse presentation-dirty signals instead of a dedicated phone-panel snapshot family.
- `gs1_godot_craft_panel_controller.h` / `gs1_godot_craft_panel_controller.cpp`: Scene-owned direct-subscriber controller for the craft/placement panel, owning the craft/placement summary label, dynamic craft-option button reconciliation, and craft-request submission while rebuilding contextual craft options and active placement preview state from `Gs1GameStateView` on coarse presentation-dirty/app-state signals and keeping only the small placement-failure cue as runtime message input.
- `gs1_godot_task_panel_controller.h` / `gs1_godot_task_panel_controller.cpp`: Direct-subscriber controller for the site task/modifier panel, owning the task-summary label and disabled task/modifier row reconciliation plus its local icon/metadata cache, now rebuilding task and modifier content from the exported gameplay-state view on coarse presentation-dirty signals instead of a dedicated task-panel snapshot family.
- `gs1_godot_regional_map_hud_controller.h` / `gs1_godot_regional_map_hud_controller.cpp`: Direct-subscriber controller for the regional-map HUD scene, owning campaign summary text, selected-site hint text, and the tech-panel entry button so future regional-map HUD buttons can be added without routing those panel-local controls through the shell, with the HUD itself simply living while the regional scene exists, applying selected-site and campaign-summary labels only when the dedicated regional-map-HUD snapshot commits or campaign resources change, never fabricating a selected site when gameplay projected none, and deriving the tech button state from surface visibility instead of the shared panel snapshot.
- `gs1_godot_regional_selection_panel_controller.h` / `gs1_godot_regional_selection_panel_controller.cpp`: Direct-subscriber controller for the regional selection panel, now rebuilding the selected-site brief, support-feed summary, provided-loadout slot grid, and deploy action directly from `Gs1GameStateView` on coarse presentation-dirty/app-state signals instead of following dedicated regional-selection snapshot or UI-panel families, with the panel still avoiding any fallback first-site selection, continuing to suppress the clear-selection button so deselection stays on blank-map click, and rendering exported support items as a read-only icon grid.
- `gs1_godot_regional_tech_tree_panel_controller.h` / `gs1_godot_regional_tech_tree_panel_controller.cpp`: Scene-owned direct-subscriber controller for the regional research overlay, owning its local progression-view state, the unlockable/technology card reconciliation, the close-button plus card-action input wiring, icon/tooltip caches, and tech action-button routing, with overlay visibility driven by authoritative progression/surface messages instead of app-state checks, the authored ladder layout now built locally from shared technology/unlock definitions while runtime progression messages only refresh dynamic per-entry state, the ladder now splitting full research-card buttons from lightweight non-interactive marker/placeholder controls so non-card rows no longer allocate either the full card subtree or button widgets on first open, and a regional-map-load prewarm pass now filling the tech and unlockable UI caches before the first panel open.
- `gs1_godot_director_control.h` / `gs1_godot_director_control.cpp`: Bootstrap `godot-cpp` `Control` that owns the shared adapter service, watches the authoritative projected app state, swaps the menu/regional/site scene instance without synthesizing bootstrap gameplay messages, and now routes the main-menu-to-regional-map transition through a lightweight loading scene while Godot threaded-loads the full regional-map `PackedScene`, with the director also owning the current explicit regional tech-tree icon preload list, driving the loading-scene progress bar from the real threaded scene/preload work, buffering/replaying all engine messages during that loading phase so controller initialization still happens only when the complete target scene, queued icon textures, and deferred engine-message batch are ready, and now applying the same async transition machinery to `SITE_LOADING` so the site session becomes active only after the site scene has instantiated and the director submits `SITE_SCENE_READY`.
- `gs1_godot_main_menu_ui_controller.h` / `gs1_godot_main_menu_ui_controller.cpp`: Main-menu UI root that owns the shell-local button wiring and start/quit runtime submissions without carrying any regional-map or site-session responsibilities.
- `gs1_godot_regional_map_scene_controller.h` / `gs1_godot_regional_map_scene_controller.cpp`: Regional-map scene root that owns world-node lookup, direct campaign-state reads through `Gs1GameStateView`, authored camp marker scene instancing, camera framing, runtime state tinting for authored marker accents, full-marker grey-out overlays for blocked sites, and regional click-picking/input dispatch while leaving backdrop art, HUD, and panel behavior to authored scenes plus dedicated UI controllers instead of generating terrain, links, or fallback marker geometry in C++, with site-marker reconciliation now rebuilt from authoritative campaign-site state on coarse presentation-dirty/app-state signals rather than a dedicated regional snapshot stream, with world picking handled from `_unhandled_input()` so Godot UI gets first crack at button presses while leftover clicks that still land inside regional-map-owned HUD/overlay bounds are swallowed before they can leak across screens, with blank-world clicks still consuming the click even when nothing was selected, unselectable locked/completed site markers staying consumed without submitting deployment-selection actions, and only submitting clear-selection when a selection actually exists while selected visuals never fall back to the first site when gameplay says nothing is selected.
- `gs1_godot_regional_map_ui_controller.h` / `gs1_godot_regional_map_ui_controller.cpp`: Regional-map UI root that owns only shell-level UI-surface visibility handling for the selection shell, summary shell, and shared tech-tree overlay while the regional scene exists, without app-state gating or panel-local subscriptions.
- `gs1_godot_site_scene_controller.h` / `gs1_godot_site_scene_controller.cpp`: Site-session scene root and world presenter that owns the bootstrap camera/light, current-camera framing for the full site board, fertility-banded textured terrain column rendering, projected tile/plant/device/worker visuals, the worker placeholder sub-scene instance, gameplay-ID-to-Godot-instance reconciliation, and direct state-view/query-driven site rendering without a separate site-view controller node, now rebuilding the site board from `Gs1GameStateView` plus `query_site_tile_view(...)` on coarse presentation-dirty/app-state signals instead of consuming the old dedicated site-scene snapshot family.
- `gs1_godot_site_session_ui_controller.h` / `gs1_godot_site_session_ui_controller.cpp`: Site-session UI root that still owns the deferred local selected-tile path, static site command wiring, and the small remaining shell-level host submissions without owning or subscribing the scene-owned status, HUD, inventory, phone, craft, task, action, overlay, summary, or tech-tree panel controllers, now reading active-site dimensions, storage state, worker-pack slots, and per-tile inspection data directly from `Gs1GameStateView` plus `query_site_tile_view(...)` instead of assembling a broad local site snapshot from runtime messages.
- `godot_progression_resources.h` / `godot_progression_resources.cpp`: Lightweight Godot-only TOML loader for progression and broader render resources, keyed by shared `tech_node_id`, unlockable content kind/id pairs, general content kind/id pairs, and domain/id entries such as modifiers, task templates, reward candidates, and sites so the adapter can resolve icon, thumbnail, and scene paths without moving shared descriptions or gameplay meaning out of the gameplay-authored content tables, now loading from the staged `res://gs1/godot_config/` package copy.
