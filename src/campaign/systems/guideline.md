# src/campaign/systems/

Campaign-level systems that react to gameplay messages and mutate only campaign-owned state.
Ownership now flows into runtime registration through compile-time shared-framework system contracts rather than per-system runtime `owned_state_sets()` overrides.

## Usage
- Read this file before scanning the folder in detail.
- If you open a child folder, read its `guideline.md` first when present.
- When files or folders in this directory change, update this file in the same change.

## Contents
- `campaign_flow_system.h`: Campaign flow system interface for high-level progression transitions, explicit ownership of app/campaign flow plus `SiteRunMeta`, direct `RuntimeInvocation`-based host/game subscription spans on the shared runtime-system interface, and compile-time subscribed-message/profile/fixed-step/emitted-runtime-message trait metadata for the typed system-pack migration, with the fully typed-enabled lifecycle family no longer needing a per-system legacy gameplay-envelope override.
- `campaign_flow_system.cpp`: Campaign flow system implementation for campaign state transitions, authoritative runtime app-state handoff, owner-resolved campaign clock deltas, narrow campaign/site lifecycle installation through `RuntimeInvocation` helpers instead of direct cross-owner split-state seeding, site-run creation through the site-run factory's split-campaign overload, direct typed handler routing for the core campaign flow messages while the runtime now translates the start/select/deploy/return gameplay-action host family before campaign-flow dispatch, typed site-run start and site-scene activation emits, adjacent-site availability promotion without duplicating already-visible prototype site markers in the revealed-site cache, and a boot-safe main-menu app-state write path that now tolerates the zero-campaign pre-bootstrap frame used by direct typed runtime tests.
- `campaign_progression_system.h`: Generic campaign progression authority interface that owns faction-progress and campaign-progression split state for scoped token balances plus threshold grants, now also declaring compile-time profile/fixed-step/emitted-runtime-message trait metadata for the runtime trait-adoption slice and no longer carrying a per-system legacy gameplay-envelope trampoline for its typed-enabled message family.
- `campaign_progression_system.cpp`: Generic campaign progression authority implementation for inline progression-event handling, reputation token mutation, one-shot threshold firing, assistant unlock mirroring, and generic `TargetGranted` emission on the migrated campaign path.
- `campaign_time_system.h`: Campaign time system interface for fixed-step-derived campaign clock advancement through the shared `RuntimeInvocation` system interface while directly owning the dedicated split `CampaignTime` state set, now also declaring explicit empty compile-time subscribed-message metadata plus profile/fixed-step/emitted-runtime-message trait metadata for the runtime system-pack migration and now leaning on the shared runtime default legacy hooks instead of local no-op message overrides.
- `campaign_time_system.cpp`: Campaign time system implementation that advances the dedicated `CampaignTime` split state directly, including derived remaining-day updates, without routing through campaign-flow-owned message indirection.
- `faction_reputation_system.h`: Campaign faction-reputation system interface retained as a compatibility placeholder while the generic campaign progression authority owns the migrated faction-reputation mutation path, now leaning on shared runtime default legacy hooks instead of local empty overrides.
- `faction_reputation_system.cpp`: Compatibility campaign faction-reputation implementation retained while the generic campaign progression authority owns the migrated faction-reputation mutation path, with the last dead local legacy-message stub removed so the placeholder stays narrowly dormant.
- `loadout_planner_system.h`: Loadout planner system interface, including direct runtime message subscription spans through the shared `RuntimeInvocation` system interface plus compile-time profile/fixed-step/emitted-runtime-message trait metadata used by the runtime system-pack migration, with deployment-site selection changes now fully relying on typed dispatch instead of a legacy gameplay-envelope override.
- `loadout_planner_system.cpp`: Loadout planner system implementation, including the content-authored baseline deployment package and content-authored per-contributor support quota, now driven from tagged runtime campaign-state access and direct typed handler routing for deployment-site selection changes.
- `regional_support_system.h`: Regional support system interface for campaign support updates.
- `technology_system.h`: Technology system interface for research/unlock progression, including direct runtime message subscription spans through the shared `RuntimeInvocation` system interface while preserving external read-only unlock query helpers and now also declaring compile-time profile/fixed-step/emitted-runtime-message trait metadata for the runtime trait-adoption slice, with the old gameplay-action host validation shim removed now that claim/refund input is translated at the runtime boundary.
- `technology_system.cpp`: Technology system implementation for content gating and the migrated tech-target grant path, still preserving compatibility total-reputation unlock helpers for baseline plants/recipes while beginning the shift toward explicit tech-node grant storage and `TargetGranted` consumption instead of owning token mutation itself, with the technology claim/refund gameplay-action host family now arriving entirely through runtime-side typed translation before the system's typed handlers validate those requests.
